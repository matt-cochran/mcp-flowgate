You and your team have a shared vocabulary. The "audit log" means a specific thing. So does "evidence pack" and "Tier 1 surface." When you write a workflow that uses one of those terms, you expect the next person reading it to mean the same thing.

The model running that workflow has no such expectation. Every session is amnesia. The term "evidence pack" in a skill body is just two English words to it.

The lexicon primitive (SPEC §30) gives you a persistent vocabulary the gateway carries across runs. Workflows reference terms by name. Agents look them up via `gateway.lexicon.lookup`. Humans (and optionally agents, with permission) extend the vocabulary via `gateway.lexicon.define`. Every in-flight workflow gets its lexicon snapshot stamped at start, so a mid-run define doesn't mutate what existing instances see.

## Declaring lexicon

A `lexicon:` block at the gateway config's top level:

```yaml
# gateway.yaml
version: "1.0.0"

lexicon:
  evidence-pack:
    definition: |
      The structured set of artefacts a workflow collects to justify
      a decision — file:line citations, test outputs, links to prior
      decisions, audit-log excerpts.
    bounded_context: governance
    examples:
      - "The vet-plan workflow assembles an evidence-pack with at
         least three independent citations."
    governance: human-only

  tier-1-surface:
    definition: |
      An interface backed by a stability commitment in STABILITY.md.
      Tier 1 surfaces never break across minor versions; deprecations
      land at major bumps with one cycle of warning.
    bounded_context: api-design
    governance: human-only
```

Each entry has:

- **`definition`** — required, free-form prose. The canonical "what this term means in this codebase."
- **`bounded_context`** — optional, names the subject area the term lives in. Lets you scope `audit-log` (governance bounded context) distinct from `audit-log` (e-commerce bounded context) without collision in the same gateway.
- **`examples`** — optional, array of usage strings. Helps the LLM see the term in context.
- **`refs`** — optional, array of links/citations.
- **`governance`** — optional, `human-only` (default) or `agent-may-propose`. See below.

Bad entries fail at config load with `INVALID_LEXICON_ENTRY`.

## Snapshot stamping

Every workflow instance carries a `_lexiconLibrary` snapshot of the entire lexicon at the moment it started. The snapshot is what template substitution reads (`{{$.lexicon.X.definition}}` in skill bodies). A mid-flight `gateway.lexicon.define` writes to the gateway's in-memory overlay; in-flight workflows are immune until they restart.

This is the same SPEC §8.2 invariant that pins `_skillsLibrary` and `_scriptsLibrary` per instance. Mid-run config edits never mutate what a running instance sees.

## Governance: human-only vs agent-may-propose

By default, every lexicon entry has `governance: human-only`. An agent calling `gateway.lexicon.define` against a `human-only` term gets `LEXICON_DEFINE_REQUIRES_HUMAN`. The workflow has to route through an `actor: human` transition to actually commit.

Switching a term to `governance: agent-may-propose` is the explicit opt-in for "this is sandbox / scratch vocabulary; agents can extend." Useful for grill-me-style discovery sessions where the agent's job is to coin terms collaboratively. Risky to default to it — once a term lands in the lexicon, every downstream workflow inherits its meaning.

## The three MCP tools

### `gateway.lexicon.search`

Search the lexicon by keyword.

**Arguments:**

| Name | Type | Required | Description |
|------|------|----------|-------------|
| `query` | string | Yes | Search terms |
| `bounded_context` | string | No | Restrict results to one bounded context |

**Returns:** array of matching entries (term + definition + governance). Useful when the model wants to see what vocabulary the operator has already declared before coining a new term.

### `gateway.lexicon.lookup`

Look up a single term by name.

**Arguments:**

| Name | Type | Required | Description |
|------|------|----------|-------------|
| `term` | string | Yes | The exact term to look up |

**Returns:** the entry (definition + governance + bounded_context + examples + refs) or a `LEXICON_TERM_NOT_FOUND` error.

### `gateway.lexicon.define`

Register a new term or update an existing one.

**Arguments:**

| Name | Type | Required | Description |
|------|------|----------|-------------|
| `term` | string | Yes | The term to define |
| `definition` | string | Yes | The canonical definition |
| `bounded_context` | string | No | Subject area scope |
| `examples` | array | No | Usage examples |
| `governance` | string | No | `human-only` or `agent-may-propose` (default `human-only`) |

**Returns:** the registered entry. Emits a `lexicon.defined` audit event on success.

**Errors:**

- `LEXICON_DEFINE_REQUIRES_HUMAN` — the caller is an agent but the term's governance is `human-only`.
- `INVALID_LEXICON_ENTRY` — the definition or shape is malformed.

The runtime overlay holds defines for the gateway's lifetime. Persist them by editing `flowgate.yaml` and reloading; otherwise they vanish on restart.

## The accumulation story

The point of the lexicon isn't to declare 100 terms up front. It's to grow vocabulary as your team and the model work together. A `grill-with-docs` workflow runs, the agent encounters "blast radius" in your codebase, asks what it means, you define it once. Every subsequent workflow inherits the definition. Three months later "blast radius" is a term every agent recognizes without you re-explaining.

That's the Eric Evans / Domain-Driven Design ubiquitous-language story, automated. Where mattpocock-style skills are stateless (vocabulary the model coined in one conversation vanishes in the next), lexicon entries accumulate. The persistent vocabulary IS the multiplier.

See `cap.research.lexicon-lookup` + `cap.summarize.lexicon-define` in [flowgate-meta](/guides/self-authoring/) for the typed cap wrappers that meta-orchestrators use to keep authored content vocabulary-consistent.

## Template substitution

Inside any skill body or workflow guidance string, `{{$.lexicon.<term>.definition}}` substitutes the snapshot's definition for that term. Missing term → standard `(unset)` stub. Workflows can build prompts that reference the operator's evolving vocabulary without re-pasting definitions.

```yaml
skills:
  plan.vet.discipline:
    body: |
      When assembling the vet packet, the agent must include:
      {{$.lexicon.evidence-pack.definition}}

      The pack is what review gates check. See {{$.lexicon.tier-1-surface.definition}}.
```

The substitution happens at instance-snapshot time. The skill body the agent reads at workflow.start carries the resolved definitions inline.

## Tier 1 stability commitment

The `gateway.lexicon.*` MCP tools and the `lexicon:` config shape are STABILITY Tier 1 — they don't break across minor versions. The `governance` default (`human-only`) is also Tier 1; flipping the default to `agent-may-propose` is a SPEC bump.
